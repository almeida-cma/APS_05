🍽️ Sistema de Pagamentos · Restaurante

Análise de requisitos e modelagem UML — quais diagramas usar?

📋 Requisitos do Sistema

📥 RF01 - Entrada

Valor total da conta (R$) em um único campo.

⚙️ RF02 - Processamento

6 formas: Dinheiro (10% off), PIX (5% off), Débito (integral), Crédito 1x(2%), 2x(4%), 3x(7%)

📤 RF03 - Saída

Exibir 6 cards com valores calculados + parcelas quando aplicável.
💡 Regras de negócio: Dinheiro = valor × 0.90 | PIX = valor × 0.95 | Débito = valor × 1.00
Crédito 1x = valor × 1.02 | Crédito 2x = (valor × 1.04)/2 | Crédito 3x = (valor × 1.07)/3
Recomendado #1 Diagrama de Caso de Uso

Por que usar? Mostra quem (ator) interage com o sistema e quais funcionalidades estão disponíveis. Aqui, o Atendente é o ator principal.

🎯 Atendente → Digitar valor → Calcular (obrigatoriamente inclui Exibir cards)
Recomendado #2 Diagrama de Atividades (Fluxograma)

Por que usar? Mostra o fluxo completo do sistema: entrada → validação → processamento paralelo das 6 fórmulas → saída.

🔀 Fluxo completo: Digitar valor → Validar (se erro exibe mensagem) → Processar 6 fórmulas → Exibir cards
Recomendado #3 Diagrama de Sequência

Por que usar? Mostra a ordem das interações entre Atendente, Sistema e Interface ao longo do tempo.

⏱️ Sequência: Atendente → Sistema → Interface → retorno → exibição dos 6 cards
Recomendado #4 Diagrama de Classes

Por que usar? Define a estrutura de dados do sistema: as classes que representam o cálculo e os resultados.

🏛️ Estrutura: CalculadoraPagamento processa o valor e gera 6 objetos ResultadoPagamento (um por forma de pagamento).
Recomendado #5 (opcional) Diagrama de Estado

Por que usar? Mostra os estados do objeto "Pagamento" durante o fluxo, útil se o sistema tiver mais regras futuras.

🔄 Estados possíveis: Aguardando valor → Validando → Processado (ou Erro se valor inválido).
📌 Resumo: Diagramas Recomendados para Este Sistema

✅ Essenciais (obrigatórios para documentar o sistema):

  • Diagrama de Caso de Uso — mostra que o Atendente é o único ator e as funcionalidades principais.
  • Diagrama de Atividades — melhor opção para representar o fluxograma completo (entrada → validação → processamento paralelo → saída).
  • Diagrama de Sequência — detalha a interação temporal entre Atendente, Sistema e Interface.

✅ Complementares (úteis para desenvolvimento futuro):

  • Diagrama de Classes — define a estrutura de dados (CalculadoraPagamento + ResultadoPagamento).
  • Diagrama de Estado — útil se o sistema evoluir para ter múltiplas etapas (ex: pagamento pendente, confirmado, cancelado).

❌ Não recomendados para este escopo: Diagrama de Componentes e Implantação (pois o sistema é simples, sem arquitetura distribuída complexa).

🧠 Conclusão: Para sistemas com regras de negócio como este restaurante, os fluxogramas mais apropriados são Diagrama de Atividades (mostra o passo a passo), Caso de Uso (mostra quem usa) e Sequência (mostra ordem das mensagens).
📐 Exercício prático: Tente desenhar você mesmo o Diagrama de Atividades no papel usando os requisitos acima!
